## 3. Matching Game architecture

### 3.1 Product modes (same engine)

| Mode | Player experience | Pairing model |
|------|-------------------|---------------|
| **Match** (asymmetric) | Two zones or free layout: match A items to B items (image↔text, text↔text, image↔image, icon↔text, etc.) | Each `MatchPair` has `left` + `right` faces (may differ by media kind) |
| **Memory** | Face-down grid; flip two; match when same `pairId` | Same pairs; both tiles show identical (or paired) backs until matched |

Authors pick **play mode** in the editor. Engine always reasons about `pairId` + two faces.

### 3.2 Content model

```ts
type MatchMediaKind = "image" | "text" | "icon" | "audio";

interface MatchFace {
  kind: MatchMediaKind;
  text?: string;
  imageUrl?: string;
  iconUrl?: string;   // emoji or /api/file URL
  audioUrl?: string;
  alt?: string;       // required for image/icon when meaningful
}

interface MatchPair {
  id: string;
  faceA: MatchFace;
  faceB: MatchFace;   // Match mode: often different kinds; Memory: often same art
  /** Memory mode: face-down art for this pair when sharedBack.enabled is false */
  back?: MatchFace;
}

type MatchingPlayMode = "match" | "memory";

interface MatchingRecord {
  gameType: "matching";
  // standard meta: title, slug, clientName, projectCode, …
  playMode: MatchingPlayMode;
  pairs: MatchPair[];
  /** Memory only: one back for all tiles when enabled; otherwise use pair.back */
  sharedBack: {
    enabled: boolean;
    face?: MatchFace;
  };
  layout: {
    columns: number | "auto";
    gapPx: number;
    tileMinPx: number;
    tileMaxPx: number;
  };
  /**
   * Visual container — not required to look like “cards”.
   * enabled ON  = optional bg, radius, border, shadow, padding.
   * enabled OFF = flat elements; assets sit without forced chrome (min hit-target still applied for a11y).
   */
  cardChrome: {
    enabled: boolean;
    backgroundHex?: string;
    borderHex?: string;
    radiusPx?: number;
    shadow?: boolean;
    paddingPx?: number;
  };
  branding: {
    backgroundHex: string;
    backgrounds: { desktop?: string; tablet?: string; mobile?: string };
    logoUrl?: string;
    logoAlign?: "left" | "center" | "right";
    showPoweredBy?: boolean;
    // board-style tokens for chrome when enabled
  };
  gameplay: {
    shuffle: boolean;
    timerSec?: number | null;
    maxAttempts?: number | null;
    mismatchDelayMs: number;
    inputModes: ("tap" | "drag" | "keyboard")[];  // MVP: all three on
  };
  intro / endScreen: /* catch pattern */;
}
```

### 3.3 Interaction (MVP includes drag)

All inputs converge on one engine:

| Input | Behaviour |
|-------|-----------|
| **Tap** | Select face → select second → resolve |
| **Drag** | Pointer/touch drag face onto target → resolve |
| **Keyboard** | Tab/arrows between tiles; Enter/Space select; Esc clear |

**Accessibility with DnD:**

- Every draggable face remains a **focusable button** (or has an equivalent keyboard path).
- Drag is **additive**, not the only path (WCAG: pointer alternative).
- Announce match/mismatch via `aria-live`.
- `prefers-reduced-motion`: skip flip/slide animations; instant settle.
- When `cardChrome.enabled === false`, keep minimum hit target (~44px) via invisible padding.

### 3.4 Content-aware sizing

- Measure face content (image intrinsic size, text metrics).
- Size tiles between `tileMinPx`–`tileMaxPx`; grid reflows.
- Memory mode may use more uniform cells; Match mode may allow asymmetric face sizes within the same pair.

### 3.5 Editor UX (content-first)

**Not:** raw pair JSON.  
**Yes:**

1. Mode toggle: **Match** | **Memory**
2. “Add pair” row: two faces with media-type picker + upload/text fields
3. Memory: per-pair **back** control; plus toggle **Use shared back for all pairs**
4. Soft warning when pair count **> 12** (“layouts can feel crowded — still allowed”)
5. Duplicate / reorder / delete pair
6. Live preview iframe (flip-cards / catch pattern)
7. **Container chrome** toggle: ON shows style controls; OFF = flat elements (no radius/shadow/card frame)
8. Branding section (shared pattern)

Elements are not required to look like cards — chrome is optional packaging around any face media.

### 3.6 Analytics / flow

| Event | Notes |
|-------|-------|
| `matching.round_start` | mode in properties |
| `matching.pair_matched` | `pairId` only — no text content |
| `matching.round_end` | moves, time, pairsMatched, completed |
| Flow outcomes | `completed`, `matching.matchedPairs`, `matching.moves`, `matching.playMode` |

## 4. Digital Board Game architecture

### 4.1 Play modes (all in scope)

| Mode | Surfaces | Session |
|------|----------|---------|
| **Solo** | `/board/{slug}` | Local |
| **Local multiplayer** | Same URL + setup screen | Memory / sessionStorage (aliases, pieces) |
| **QR play-along** | `/board/{slug}/host`, `/join/{code}`, `/present/{code}` | Generalized game session (from quiz) |

QR influences architecture from day one: **host-authoritative** turns, shared session blob, participant icons, lobby lock, poll-by-revision.

### 4.2 Graph-capable board model (MVP = perimeter)

```ts
interface BoardNode {
  id: string;
  label?: string;
  colorHex?: string;
  iconUrl?: string;
  actionRef: BoardActionRef;   // registry reference — not inline hardcoded behaviour
  /** Visual / path position for editor + renderer */
  position: { x: number; y: number };  // normalized 0–1 or design coords
}

interface BoardEdge {
  id: string;
  fromNodeId: string;
  toNodeId: string;
  label?: string;
  /** MVP: unused; future branching conditions */
  condition?: BoardEdgeCondition;
}

interface BoardGraph {
  nodes: BoardNode[];
  edges: BoardEdge[];
  startNodeId: string;
  /** MVP helper: ordered ring derived from edges for perimeter render */
  layoutHint: "perimeter" | "free";  // MVP ships perimeter
}

interface BoardWinCondition {
  type: "laps" | "reach_node" | "points" | "custom";
  lapsRequired?: number;
  finishNodeId?: string;
}
```

**MVP renderer:** treat graph as a **cycle** (perimeter loop) when `layoutHint === "perimeter"`.  
**Future:** free layout + branching edges without migrating node/action data.

### 4.3 Action Registry (like component registry)

```ts
/** Platform registry — extend without changing board engine */
interface BoardActionDefinition {
  actionType: string;           // e.g. "none", "message", "moveRelative", "launchComponent"
  label: string;
  category: "basic" | "movement" | "content" | "launch" | "custom";
  /** Schema for editor form fields */
  configFields: BoardActionField[];
  /** MVP: which handlers are implemented in runtime */
  status: "shipped" | "planned";
}

interface BoardActionRef {
  actionType: string;
  config: Record<string, unknown>;
}
```

**MVP shipped actions (examples):**

| actionType | Config | Effect |
|------------|--------|--------|
| `none` | — | Land only |
| `message` | `text` | Show modal / centre copy |
| `moveRelative` | `delta: number` | ±N steps along **current edge choice** (MVP: single outbound) |
| `skipTurn` | — | Next player |
| `extraTurn` | — | Same player again |
| `gotoNode` | `nodeId` | Teleport |

**Planned (register now, implement later):**

| actionType | Config | Effect |
|------------|--------|--------|
| `launchComponent` | `moduleType`, `moduleInstanceId`, `returnMode` | Open quiz / wheel / catch / form / certificate in overlay or route; resume board on complete |
| `drawCard` | `deckId` | Future decks |
| `awardPoints` | `points` | Scoring boards |

Engine loop:

```text
land on node → resolve actionRef via registry handler → apply BoardEffect[] → continue
```

Unknown / `planned` actions: no-op + Studio warning — **never crash**.

### 4.4 Visual board editor

Primary UX:

1. Canvas shows perimeter (or free) board preview  
2. **Click tile** → side panel: label, colours, icon, **action type + config**  
3. Add / remove tiles with “insert on ring” controls  
4. Recommended count guidance (e.g. soft warn outside 16–48)  
5. Piece icon set upload (quiz pattern)  
6. Branding: board fill, tile border, centre panel, breakpoint BGs  

Avoid making “edit JSON list of 40 tiles” the default path (advanced collapse OK).

### 4.5 QR session (generalize quiz early)

Extract shared kit while building board (don’t wait for retrofit):

```text
game-session store:
  code, hostKey, revision, expiresAt, lobbyOpen
  gameType: "board-game"
  gameId, gameSlug
  participants: [{ id, icon, seat }]   // alias optional, presentation-only, not tracked
  phase: lobby | playing | ended
  + board-specific payload (positions, turn, lastRoll)
```

APIs (mirror quiz):

- `POST/GET /api/board-session` (or generic `/api/game-session`)
- `POST /api/board-join`
- `POST /api/board-control` (host: start, force next, end)
- `POST /api/board-action` (player intent: roll / ack — server/host applies)

**Authority:** Host or server applies dice + moves; clients send intents.

**Aliases:** Allowed on join/setup UI; stripped from `track()`; not written to measurement DB.

### 4.6 Local multiplayer setup

Facilitator screen before play:

- Player count  
- Aliases (session-only)  
- Piece selection from icon set  

Same rules: aliases never tracked.

## 8. Risks and mitigations

| Risk | Mitigation |
|------|------------|
| DnD + a11y conflict | Keyboard parity mandatory; drag additive; live regions |
| Blob races on QR turns | Host/server authority; intent queue; revision checks |
| Visual board editor scope creep | MVP = perimeter only; free layout later |
| Action Registry overbuild | Ship 5–6 handlers; register planned launch types as stubs |
| Alias leakage into analytics | Strip in track helpers; review ingest validation |
| Cramped boards at high tile counts | Soft editor warnings; responsive tile shrink |
