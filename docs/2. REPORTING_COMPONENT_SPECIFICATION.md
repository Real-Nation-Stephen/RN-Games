# Reporting Component Specification

**Document status:** Draft for implementation review  
**Platform:** Real Nation Digital Platform  
**Depends on:** `TRACKING_AND_MEASUREMENT_SPECIFICATION.md`  
**Source baseline:** `COMPONENT_DATA_EVENT_INVENTORY.md`, repository `main` after commit `1ab70ac` (July 2026)  
**Primary audience:** Product, development, reporting, client services, design and compliance

---

## 1. Purpose

This specification defines the Reporting Component for the Real Nation Digital Platform.

The Reporting Component turns the platform’s shared measurement data into:

- configurable dashboards;
- client-facing report views;
- internal operational views;
- filtered exports;
- clear explanations of what each metric means;
- reusable report templates for courses, flows and standalone components.

The Reporting Component does **not** create a second tracking system and does not duplicate event collection.

It reads from the shared Measurement layer defined in `TRACKING_AND_MEASUREMENT_SPECIFICATION.md`.

The guiding principle is:

> **Tracking records what happened. Reporting decides how that information is presented, interpreted and shared.**

---

## 2. Relationship to the Measurement Component

The Reporting Component is the presentation and configuration layer over the Measurement Component.

Recommended architecture:

```text
Course / Flow / Standalone Component
            │
            ▼
   Measurement Component
   - defines scope
   - collects events
   - stores fields
   - applies retention
            │
            ▼
    Reporting Component
   - selects metrics
   - defines labels
   - builds dashboard
   - applies filters
   - controls access
   - creates exports
```

For the first implementation, Measurement and Reporting may appear as one combined admin component:

> **Measurement & Reporting**

However, the code and specifications should keep the responsibilities distinct so that:

- reporting can evolve without changing event collection;
- multiple report views can use one measurement scope;
- client-facing views can differ from internal views;
- metrics can be hidden without stopping collection;
- future reporting templates can be reused across deployments.

## 4. Reporting Component scope

A Reporting Component may attach to:

1. a Course;
2. a Flow;
3. a standalone component deployment.

The scope determines which data is included.

### 4.1 Course scope

A report attached to a Course includes:

- course-level activity;
- all flows nested inside the course;
- all components nested directly or indirectly inside the course;
- only activity generated within that course context.

### 4.2 Flow scope

A report attached to a Flow includes:

- flow-level activity;
- all components nested inside the flow;
- only activity generated within that flow context.

### 4.3 Standalone scope

A report attached to a standalone component includes:

- direct activity for that deployment only;
- no activity from the same reusable component when used elsewhere.

### 4.4 Scope isolation example

```text
Dino Dash template
├── Standalone ESB campaign deployment
├── Digital 404 course deployment
└── Internal test deployment
```

Each deployment must produce separate reporting results.

A report attached to Digital 404 must not include direct plays of the ESB or test versions, even if all deployments use the same Runner template.

## 5. Reporting Component objectives

The Reporting Component should:

- make useful results visible without exposing unnecessary technical data;
- create consistent reporting across all component types;
- allow client-friendly labels;
- support aggregate reporting by default;
- allow internal users to inspect more detail where authorised;
- explain metric definitions clearly;
- make trends, comparisons and drop-off points easy to understand;
- generate reusable report links;
- support secure exports;
- display compliance and retention status;
- remain useful after identifiable records have been purged;
- reduce manual report-building effort.

## 6. Reporting principles

### 6.1 Outcomes before infrastructure

Client-facing reports should prioritise information such as:

- participation;
- engagement;
- completion;
- learning outcomes;
- return behaviour;
- conversion;
- game performance.

They should not lead with:

- session IDs;
- internal component UUIDs;
- schema versions;
- database fields;
- raw event names.

### 6.2 Presentation does not alter collection

Changing a report label or hiding a metric must not change the underlying event schema.

### 6.3 Aggregate by default

New report views should default to aggregate metrics.

Participant-level views should require:

- explicit enablement;
- an appropriate access role;
- compliance approval;
- a valid reporting purpose.

### 6.4 No silent ambiguity

Every metric must have:

- a display label;
- a plain-English definition;
- a calculation rule;
- a unit;
- an applicable scope;
- any known limitations.

### 6.5 Reports are deployment-specific

Metrics should always be calculated within the Reporting Component’s attached scope unless a user deliberately creates a multi-deployment portfolio view.

### 6.6 Reports must remain readable

Not every supported metric should be shown.

The report builder should recommend a concise, useful default set based on the attached scope and component types.

## 8. Report audiences and views

### 8.1 Internal Admin View

Purpose:

- validate tracking;
- inspect configuration;
- diagnose missing events;
- review raw and aggregate data;
- monitor retention;
- manage report access.

May include:

- raw event counts;
- schema warnings;
- component identifiers;
- session-level drill-down;
- test-data controls;
- purge status.

### 8.2 Client View

Purpose:

- communicate performance clearly;
- show outcomes;
- support decision-making;
- avoid unnecessary technical detail.

Should include:

- selected headline metrics;
- trends;
- comparisons;
- charts;
- plain-English insights;
- methodology notes;
- export options approved for the client.

### 8.3 Facilitator or Teacher View

Purpose:

- track class or cohort participation;
- view learning outcomes;
- identify content gaps;
- avoid unnecessary exposure of learner data.

May include:

- enrolments;
- completion by class;
- average score;
- question accuracy;
- progress distribution;
- learners requiring follow-up, where permitted.

### 8.4 Public Aggregate View

Purpose:

- display live or campaign-level results safely.

May include:

- total plays;
- public leaderboard;
- poll totals;
- average score;
- completion milestones.

Must not expose:

- email;
- raw names unless intentionally public and moderated;
- session IDs;
- individual response histories;
- hidden or rejected content.

### 8.5 Learner Personal Summary

Purpose:

- show an individual their own progress and outcomes.

May include:

- completed items;
- quiz results;
- badges;
- certificates;
- personal progress;
- remaining items.

Must only show the authenticated or resume-linked learner’s own data.

## 9. Dashboard structure

A default report should use the following hierarchy.

### 9.1 Report Header

- report title;
- attached course, flow or component;
- reporting period;
- last updated time;
- client/project name;
- project/design code where internal;
- compliance status indicator;
- export button.

### 9.2 Executive Summary

A short generated or manually written overview containing:

- participation;
- completion;
- strongest outcome;
- primary challenge/drop-off;
- notable trend.

Example:

> 428 people started the experience and 81% completed it. Replay behaviour was strongest in Dino Dash, while the final quiz showed a 24-point improvement from baseline.

Generated summaries should be editable and clearly marked as system-generated until reviewed.

### 9.3 Headline Metrics

Recommended maximum: 4–8 cards.

Examples:

- Total Participants
- Completion Rate
- Average Engagement Time
- Returning Participants
- Average Score
- Conversion Rate
- Certificates Earned
- Feedback Rating

### 9.4 Trends

Time-based views such as:

- participation by day;
- participation by hour;
- completion over time;
- return visits;
- score trends;
- conversion trends.

### 9.5 Journey or Funnel

For Courses and Flows:

```text
Viewed
→ Started
→ Reached midpoint
→ Completed
→ Returned
```

For a Flow:

```text
Step 1
→ Step 2
→ Step 3
→ End
```

### 9.6 Component Performance

A breakdown for each nested component.

Example:

| Component | Views | Starts | Completion | Avg. Time | Key Outcome |
|---|---:|---:|---:|---:|---|
| Welcome page | 428 | 420 | 98% | 1m 12s | 89% CTA rate |
| Learning survey | 412 | 398 | 92% | 2m 04s | 7.2 avg confidence |
| Dino Dash | 390 | 376 | 86% | 3m 41s | 2.3 plays/player |
| Rookie badge | 331 | 331 | 100% | — | 331 earned |

### 9.7 Learning Outcomes

Where supported:

- baseline score;
- final score;
- improvement;
- delayed recall;
- confidence change;
- question accuracy;
- hardest topics;
- class comparison.

### 9.8 Feedback

- average rating;
- response count;
- key selection distributions;
- open-text response count;
- export link for authorised users.

Open-text content should not be displayed directly in aggregate dashboards by default.

### 9.9 Technical and Compliance Footer

- data source;
- metric definitions;
- report generated time;
- retention summary;
- privacy note;
- report access classification.

## 15. Insight annotations

Reports should support manual annotations.

Examples:

- “The campaign email was sent on 12 March.”
- “The event moved venue on day two.”
- “Question 4 wording was updated.”
- “The leaderboard was reset between sessions.”

Annotations should appear on time-series charts and in exports.

This prevents incorrect interpretation of changes caused by real-world events.

---

## 16. Generated insights

The platform may generate plain-English observations from approved aggregate metrics.

Examples:

- “Participation peaked between 14:00 and 15:00.”
- “Returning players achieved an average score 18% higher than first-time players.”
- “Most learners dropped off before the final quiz.”
- “Question 6 had the lowest accuracy.”
- “Character B was selected twice as often as Character A.”

Generated insights must:

- use aggregate data;
- cite the metric and period;
- avoid legal, causal or psychological claims;
- be editable;
- be labelled as generated until approved;
- avoid identifying individuals.

| Learners | Team Members |
| Course Items | Lessons |
| Flow | Journey |
| Completion | Finished |

## 19. Date handling and freshness

Every report should show:

- reporting timezone;
- data range;
- last successful refresh;
- latest event timestamp;
- whether data is live or delayed;
- any incomplete period.

The platform should avoid comparing a partial current day/week with a complete previous period without warning.

---

## 20. Comparisons

Supported comparisons may include:

- previous period;
- previous event day;
- previous deployment version;
- class/cohort;
- component;
- first-time versus returning;
- baseline versus final;
- target versus actual.

The report must state the comparison basis.

No comparison should appear if sample size or data quality is too weak.

---

## 21. Small-number handling

To reduce re-identification risk and misleading interpretation:

- suppress or combine very small cohort counts where appropriate;
- do not expose identifiable drill-down from small aggregate groups;
- warn where percentages are based on very low counts;
- allow the Compliance Component to define a minimum group threshold.

Example:

> Results hidden because this group contains fewer than the configured minimum number of participants.

## 24. Live versus snapshot reports

### Live report

- updates as data arrives;
- shows current status;
- suitable for ongoing courses and events.

### Snapshot report

- fixed at a selected time;
- preserves the exact figures used in a presentation or final report;
- should record report version and generation time.

A snapshot must not silently change after creation.

## 26. Compliance integration

The Reporting Component must read status from the linked Compliance Component.

Display states:

- **Ready**
- **Ready with recommendations**
- **Review required**
- **High-risk issue**
- **Configuration incomplete**

The report builder should warn when:

- a requested field is not approved for reporting;
- a personal field is being added to a client report;
- a cohort is too small;
- retention is missing;
- the report link would expose personal data publicly;
- consent or privacy wording is inconsistent with actual reporting;
- the underlying deployment contains unresolved compliance warnings.

The Reporting Component does not make final legal decisions.

It enforces configured rules and surfaces issues for review.

---

## 27. Retention integration

The report should show:

- identifiable-data retention period;
- raw-event retention period;
- aggregate retention status;
- next purge date;
- records scheduled for purge;
- last completed purge;
- whether a snapshot includes data that will later be deleted.

After personal data is purged:

- aggregate report cards should remain where valid;
- individual drill-down should disappear;
- exports should reflect the current access and retention state;
- archived snapshots containing personal data must follow their own retention rules.

---

## 28. Data quality and coverage

The report builder should show a Data Quality panel.

Checks include:

- tracking coverage by component;
- missing view/start/complete events;
- unknown deployment context;
- duplicate events;
- conflicting completion signals;
- missing session IDs;
- late-arriving data;
- deprecated event names;
- test traffic included;
- incomplete migration data.

Client reports should not quietly display unreliable metrics.

A metric with incomplete coverage should show:

> Data incomplete — 82% of eligible sessions emitted the required event.

---

## 29. Default empty and low-data states

Examples:

### No data

> No activity has been recorded for this reporting period yet.

### Insufficient data

> There is not enough information to calculate this metric reliably.

### Metric unavailable

> This component does not currently emit the events required for this metric.

### Disabled

> Collection for this metric is disabled in the linked Measurement Component.

### Purged

> Identifiable records for this period have been deleted in line with the configured retention policy. Aggregate results remain available.

## 33. Implementation acceptance criteria

The Reporting Component is ready when:

- it reads from the shared measurement store rather than duplicating tracking;
- it can attach to a Course, Flow or standalone deployment;
- nested component activity is included correctly;
- direct and nested uses of the same component remain separate;
- aggregate client reports hide internal identifiers;
- metrics can be enabled, hidden and relabelled;
- dashboard templates produce useful default views;
- metric definitions are visible;
- filters apply consistently;
- report links support access controls;
- exports respect role and compliance settings;
- compliance and retention status are visible;
- personal drill-down is permission-controlled;
- data-quality warnings are surfaced;
- reports remain valid after personal-data purge where aggregates survive;
- Course, Flow and Component terminology is used consistently.

## 34. Decisions requiring implementation review

## 35. Recommended product decision

For simplicity in the first release:

- create one admin item called **Measurement & Reporting**;
- attach it to a Course, Flow or standalone deployment;
- use one shared setup for collection scope, report configuration and retention;
- expose separate internal tabs for Measurement and Reporting;
- allow one secure client report link;
- keep the data model flexible enough for multiple report views later.

This gives administrators one obvious place to configure results without creating duplicate tracking or asking them to understand the underlying architecture.

## 36. Source findings carried forward

This specification responds directly to the current platform limitations identified in the component inventory:

- course activity is not emitted through the shared analytics layer;
- standalone, flow and course contexts are not consistently available on events;
- many components only emit completion and lack view/start events;
- reporting is split across Blob state, track logs, Google Sheets and component-specific exports;
- there is no central report builder;
- labels cannot currently be customised consistently;
- there is no secure reusable report-view layer;
- personal, pseudonymous and aggregate data are not cleanly separated;
- retention and deletion status are not visible in reports;
- existing metrics often require manual joins or inference.

The Tracking & Measurement Specification defines the future data foundation.

This document defines how that foundation should become a useful, safe and client-ready reporting product.
