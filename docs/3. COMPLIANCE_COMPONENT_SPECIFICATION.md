# Compliance Component Specification

**Document status:** Draft for implementation review  
**Platform:** Real Nation Digital Platform  
**Depends on:** `TRACKING_MEASUREMENT_SPECIFICATION.md`, `REPORTING_COMPONENT_SPECIFICATION.md`  
**Source baseline:** `COMPONENT_DATA_EVENT_INVENTORY.md`, repository `main` after commit `1ab70ac` (July 2026)  
**Primary audience:** Product, development, compliance, reporting, client services and administrators

---

## 1. Purpose

This specification defines the Compliance Component for the Real Nation Digital Platform.

The Compliance Component must inspect a deployment’s actual configuration and produce an implementation-level compliance assessment, recommendations, publish status and documentation outputs.

It must:

- identify the fields, events, media and identifiers a deployment can process;
- distinguish technically supported data from data enabled for the current deployment;
- consume the same metadata used by Measurement and Reporting;
- detect privacy and data-governance risks through deterministic rules;
- recommend safer configuration choices without presenting recommendations as legal decisions;
- generate deployment-specific documentation from structured facts;
- validate compliance configuration before publish;
- record administrator overrides and subsequent configuration changes;
- expose retention, reporting and publication risks in one place.

The Compliance Component must not require administrators to manually describe information the platform already knows.

This document does not provide legal advice, determine lawful basis, or author privacy-policy wording. It defines the system behaviour required to support responsible configuration and review.

## 2. Component Role and Scope

### 2.1 Role

The Compliance Component is a non-participant-facing foundation Component attached to a deployment.

It may attach to:

- a Course;
- a Flow;
- a standalone Component deployment.

It inspects the attached scope and all nested Components included in that deployment.

### 2.2 Scope behaviour

When attached to a Course, the Compliance Component must inspect:

- Course-level session and identity behaviour;
- every direct Course item;
- every nested Flow;
- every Component instance within those Flows;
- attached Measurement configuration;
- attached Reporting configuration;
- configured retention policies;
- publication and access settings.

When attached to a Flow, it must inspect:

- Flow-level session behaviour;
- every nested Component instance;
- Flow-level Measurement, Reporting and retention configuration;
- any inherited platform or client defaults.

When attached to a standalone Component, it must inspect only that deployment and its related Measurement, Reporting, retention and publication settings.

### 2.3 One assessment per deployment

Compliance status must be deployment-specific.

The same Component definition may receive different assessments when used in different contexts. For example:

- a Form collecting an email address in an internal adult training Course;
- the same Form collecting an email address in a public children’s event;
- the same Form configured without identifiable fields.

These must not share one global compliance result.

### 2.4 Recommended product structure

For the ESB pilot, expose Compliance as a dedicated section within the deployment administration area and connect it directly to the existing **Measurement & Reporting** setup.

The user interface may present these as separate tabs, but they must share one underlying deployment context and metadata source.

## 3. Core Implementation Principles

### 3.1 Detect first, ask second

The system must derive all facts that can be determined from:

- Component metadata contracts;
- deployed Component configuration;
- Measurement collection settings;
- Reporting visibility and export settings;
- retention configuration;
- access and publication configuration;
- Course, Flow and deployment context;
- stored controller and processor defaults.

Administrators should only enter information that the platform cannot determine reliably.

### 3.2 One fact, one source of truth

The Compliance Component must not maintain a duplicate manually entered inventory of fields or events.

Where a source fact changes, the compliance assessment must be regenerated from the source configuration.

### 3.3 Recommendations are not legal decisions

Rule outputs must use language such as:

- `Expected`;
- `Recommendation`;
- `Warning`;
- `Review required`;
- `High risk`.

The UI and generated documents must state that outputs are configuration guidance and do not replace legal review.

### 3.4 Deterministic and auditable

Every compliance finding must record:

- rule identifier;
- rule version;
- severity;
- source facts evaluated;
- affected scope and field;
- generated message;
- recommendation;
- timestamp;
- resolution status;
- administrator override, where applicable.

### 3.5 Changes invalidate stale assessments

A compliance assessment must be marked stale whenever a relevant source changes, including:

- a Component is added, removed or reconfigured;
- a field is enabled or disabled;
- Measurement settings change;
- report visibility or exports change;
- audience, country or deployment purpose changes;
- retention changes;
- public access changes;
- controller, processor or privacy contact changes.

Publish validation must always use a current assessment.

## 4. Automatic Deployment Scanning

### 4.1 Scan inputs

The scanner must consume:

1. deployment metadata;
2. Course and Flow structure;
3. every deployed Component instance and version;
4. each Component’s shared metadata contract;
5. enabled field configuration;
6. enabled event and outcome configuration;
7. Measurement collection states;
8. Reporting fields, filters, drill-downs and exports;
9. public/private access settings;
10. retention policies and purge dates;
11. deployment configuration entered under Section 7;
12. manual classification overrides;
13. previous compliance overrides.

### 4.2 Scan output

The scan must create a normalised deployment inventory containing:

- detected Components;
- detected fields;
- detected events;
- derived outcomes and metrics;
- identity and session identifiers;
- media inputs and outputs;
- public exposure paths;
- report exposure paths;
- retention state;
- missing metadata;
- unresolved manual configuration.

### 4.3 Field states

For each detected field, the scanner must distinguish:

| State | Meaning |
|---|---|
| `supported` | The Component can process the field. |
| `enabled` | The deployment is configured to collect or generate it. |
| `collected` | Measurement will store it. |
| `reported` | Reporting may display or export it. |
| `published` | It may appear on a public or externally accessible surface. |
| `retained` | It remains stored for a configured period. |
| `transient` | It is processed but not retained after the immediate operation. |

Rules must evaluate the applicable state rather than assuming all supported fields are collected.

### 4.4 Scan lifecycle

Run the scanner:

- when the Compliance section is first opened;
- when an administrator selects **Rescan**;
- after relevant configuration changes;
- before documentation generation;
- before Publish validation;
- after deployment duplication;
- after platform metadata or rule versions change.

### 4.5 Unknown metadata

If a Component lacks a valid metadata contract, the scanner must:

- mark the Component as `metadata_incomplete`;
- generate a Review Required finding;
- prevent a Ready status;
- identify the missing contract fields;
- require implementation review before publication unless an authorised administrator overrides.

## 6. Data Classification

### 6.1 Required categories

The scanner must automatically classify detected fields and records into one or more of the following categories:

| Category | Definition for platform use | Examples |
|---|---|---|
| **Anonymous** | Cannot reasonably be linked to an identifiable person within the platform or through available auxiliary data. | Aggregate completion count with no session linkage. |
| **Aggregate** | Summary statistic derived from multiple records. It may or may not be anonymous depending on group size and source linkage. | Average score, daily starts. |
| **Pseudonymous** | Linked through an identifier that does not directly name the participant but can distinguish or reconnect records. | `participantId`, `sessionId`, resume token, external leaderboard ID. |
| **Personal Data** | Directly or indirectly identifies a person. | Name, email, phone, identifiable alias, IP address. |
| **Special Category** | Field intended to collect or infer special-category information. | Health information, ethnicity, religion, biometric identification. |
| **Media** | Uploaded, recorded or generated image, audio, video or document content. | Pinboard photo, recorded answer, certificate file. |
| **Behavioural** | Describes actions, choices, progress, performance or engagement. | Quiz answers, score, completion, CTA click, dwell time. |
| **Technical** | Device, network, application or diagnostic information. | IP rate-limit key, browser, error log, device type. |
| **Children’s Data** | Data relating to a child or collected in a deployment configured for children. This is an additional contextual classification. | Child learner name, school/class cohort, child quiz response. |

A field may have multiple classifications. For example, a child’s quiz response may be `behavioural`, `pseudonymous` and `children_data`.

### 6.2 Classification sources

Classification priority must be:

```text
Component contract default
→ deployment context augmentation
→ deterministic rule augmentation
→ authorised manual override
```

The platform must retain both the original automatic classification and the effective classification.

### 6.3 Context augmentation

The deployment context may add classifications without replacing the base classification.

Examples:

- target audience includes children → participant-linked fields also receive `children_data`;
- a report publishes session-level scores → `pseudonymous` remains applicable even when direct names are hidden;
- a form label is changed from “Comments” to “Medical requirements” → a special-category review rule is triggered;
- an uploaded photo is displayed in a public gallery → `media` plus publication-risk rules apply.

### 6.4 Manual override

An authorised administrator may override a classification where justified.

Every override must require:

- selected replacement or additional classification;
- mandatory justification;
- administrator identity;
- timestamp;
- affected field and deployment;
- previous value;
- expiry or review date where appropriate.

Manual override must not delete the automatic classification record. It changes the effective value used by the rule engine.

### 6.5 Special Category handling

When a field is classified as Special Category:

- status cannot be Ready without explicit review completion;
- the purpose and access restrictions must be provided;
- retention must be configured;
- public reporting must default to disabled;
- exports containing the field must default to disabled;
- the finding must remain at least Review Required unless a future approved rule explicitly allows an Expected scenario.

## 7. Deployment Configuration

### 7.1 Required configuration

The administrator configures the following facts where they cannot be inherited from an approved client or platform profile:

| Field | Required | Behaviour |
|---|---:|---|
| Controller | Yes for external deployment | Select existing organisation or enter structured organisation details. |
| Processor | Yes | Default to Real Nation or configured platform operator; editable only by authorised roles. |
| Joint Controllers | Conditional | Zero or more structured organisations. |
| DPO / privacy contact | Yes for external deployment | Inherit from controller profile where available. |
| Privacy Policy URL | Conditional | Required when linking to an external controller policy; otherwise generated hosted page may be used. |
| Target country/countries | Yes | One or more jurisdictions. Ireland is not assumed automatically for client deployments. |
| Target audience | Yes | Adult, mixed, children, employees, students, public, invited group or custom tags. |
| Internal / External | Yes | Determines documentation and publication expectations. |
| Education | Boolean/context tag | Activates education-specific recommendations. |
| Events | Boolean/context tag | Activates event and kiosk rules. |
| Marketing | Boolean/context tag | Activates marketing and communication rules. |
| Healthcare | Boolean/context tag | Activates health and special-category review rules. |
| Children | Boolean/context tag | Activates children’s-data rules. |

### 7.2 Inheritance

Recommended inheritance order:

```text
Platform defaults
→ client/organisation profile
→ project/deployment profile
→ explicit deployment override
```

Inherited values must show their source and allow authorised override.

### 7.3 Structured organisation records

Controller, Processor and Joint Controller data must use structured organisation records rather than free text.

Minimum fields:

- organisation name;
- organisation role;
- registered or business address where supplied;
- privacy contact name or role;
- privacy contact email;
- DPO contact, where applicable;
- privacy-policy URL;
- organisation record version.

### 7.4 Configuration validation

The system must flag:

- missing Controller for external deployments;
- missing Processor;
- missing privacy contact;
- invalid privacy-policy URL;
- children enabled without audience details;
- healthcare enabled without review owner;
- target countries not covered by the current rule set;
- contradictory settings, such as `internal=true` with a public anonymous URL.

## 8. Best Practice Recommendation Engine

### 8.1 Engine type

Implement a deterministic, versioned rule engine.

The engine evaluates normalised facts produced by the scanner and deployment configuration. It must not depend on free-form AI interpretation for MVP status decisions.

Each rule must define:

```ts
type ComplianceRule = {
  id: string;
  version: number;
  title: string;
  description: string;
  appliesWhen: RuleExpression;
  severity: "expected" | "recommendation" | "warning" | "review_required" | "high_risk";
  messageTemplate: string;
  recommendationTemplate?: string;
  affectedScopes: string[];
  blocking: boolean;
  overridable: boolean;
  documentationTags: string[];
};
```

### 8.2 Rule inputs

Rules may evaluate:

- deployment audience and context tags;
- detected fields and classifications;
- collection state;
- public/private access;
- Reporting visibility and export settings;
- retention presence and duration;
- controller and processor configuration;
- identity method;
- moderation configuration;
- consent/privacy-gate configuration;
- target country;
- Component combinations;
- missing or incomplete metadata.

### 8.3 Rule output

Each rule result must include:

- result ID;
- rule ID and version;
- outcome severity;
- plain-language message;
- recommended action;
- exact source facts;
- affected field, Component and deployment;
- whether it blocks publish;
- whether it may be overridden;
- resolution method;
- created and last-evaluated timestamps.

### 8.4 Required MVP rules

The following rules are required for the ESB pilot.

| Rule ID | Condition | Result | Default recommendation |
|---|---|---|---|
| `CHILD_EMAIL_001` | Children enabled and email collection enabled | Warning | Consider Resume Links or another non-email return method. |
| `CHILD_DIRECT_ID_002` | Children enabled and direct name/phone/email collected without a documented need | Review required | Remove the field or document why it is necessary. |
| `LEADERBOARD_REAL_NAME_001` | Leaderboard displays participant-entered real names | Recommendation | Use moderated aliases or generated display names. |
| `LEADERBOARD_PUBLIC_ID_002` | Public leaderboard exposes names or stable participant identifiers | High risk | Disable public identity exposure or switch to aliases. |
| `PHOTO_PUBLIC_001` | Photo upload enabled and content may appear in a public gallery | Review required | Confirm publication consent and moderation workflow. |
| `PHOTO_NO_MODERATION_002` | Public or shared media surface has no moderation setting | High risk | Enable pre-publication moderation or disable publication. |
| `CERT_NAME_001` | Certificate uses learner name and is delivered only to that learner | Expected | No warning. Record only required generation/download events. |
| `RETENTION_MISSING_001` | Any retained personal, pseudonymous, behavioural or media record lacks a retention policy | High risk | Configure a retention period and purge action. |
| `PUBLIC_REPORT_NAME_001` | Public report includes learner names | High risk | Remove learner-level identity from the public report. |
| `PUBLIC_REPORT_SMALL_002` | Public report exposes small groups below configured threshold | Warning | Suppress or combine small-number results. |
| `SPECIAL_CATEGORY_001` | Special Category field is enabled | Review required | Confirm purpose, access, retention and review ownership. |
| `SPECIAL_PUBLIC_002` | Special Category data is reported or published externally | High risk | Disable external exposure pending formal review. |
| `FORM_FREE_TEXT_001` | Free-text field enabled | Recommendation | Add guidance not to submit unnecessary personal or sensitive information. |
| `RESUME_TOKEN_001` | Course resume uses opaque token without direct identity | Expected | No warning where token security and expiry are configured. |
| `EMAIL_MARKETING_001` | Email field collected and Marketing enabled | Review required | Confirm communication purpose and opt-in configuration. |
| `EMAIL_SIGNUP_OPTIN_002` | Email Signup has no explicit opt-in state | High risk | Add an explicit opt-in and record its wording/version. |
| `REPORT_EXPORT_PII_001` | Personal data included in downloadable report export | Review required | Restrict export role, fields and expiry. |
| `REPORT_PUBLIC_TECH_002` | Public report exposes technical identifiers | High risk | Remove internal and technical identifiers. |
| `IP_RETENTION_001` | IP address retained beyond operational rate limiting | Review required | Store only where required, restrict access and configure short retention. |
| `NO_CONTROLLER_001` | External deployment lacks Controller | Review required | Configure the responsible Controller. |
| `NO_PRIVACY_CONTACT_001` | External deployment lacks privacy contact | Review required | Add DPO or privacy contact details. |
| `NO_PRIVACY_PAGE_001` | External deployment collects personal or pseudonymous data and has no hosted or linked privacy page | High risk | Configure and publish the deployment privacy page. |
| `UNSUPPORTED_COUNTRY_001` | Target country has no active ruleset | Review required | Obtain jurisdiction-specific review before publish. |
| `METADATA_GAP_001` | Deployed Component lacks complete compliance metadata | Review required | Complete the Component metadata contract. |
| `OVERRIDE_STALE_001` | A previous override applies to changed source facts | Review required | Reconfirm or remove the override. |
| `PURGE_OVERDUE_001` | Retention purge date has passed and records remain | High risk | Run or investigate the purge job. |
| `PUBLIC_FORM_001` | Public form collects personal data without an active privacy notice | High risk | Add the privacy notice before submission. |
| `CHILD_PUBLIC_MEDIA_001` | Children enabled and child-linked media may be public | High risk | Disable public display pending safeguarding and publication review. |

### 8.5 Rule examples

```text
Children + Email collection
→ Warning
→ “Consider Resume Links instead.”
```

```text
Leaderboard + Real names
→ Recommendation
→ “Aliases are recommended.”
```

```text
Photo upload + Public gallery
→ Review required
→ “Ensure publication consent and moderation are configured.”
```

```text
Certificate + Learner name + Private learner delivery
→ Expected
→ No warning
```

```text
Retained data + No retention policy
→ High risk
```

```text
Public report + Learner names
→ High risk
```

### 8.6 Rule precedence

Where multiple rules apply to the same issue:

1. retain all individual findings for audit;
2. show the highest severity prominently;
3. group related findings in the UI;
4. do not suppress a High Risk result because a lower-severity rule also applies.

### 8.7 Resolution actions

A finding may be resolved by:

- changing the source configuration;
- adding missing required configuration;
- applying a safer recommended option;
- completing a required review task;
- recording an authorised override.

The engine must re-evaluate after each resolution.

## 9. Compliance Status

### 9.1 Status values

Every deployment must have one effective status:

| Status | Trigger |
|---|---|
| 🟢 **Ready** | Current scan completed; no unresolved Recommendation, Warning, Review Required or High Risk findings; all mandatory deployment details and retention settings are complete. `Expected` results may remain. |
| 🟡 **Ready with recommendations** | No unresolved High Risk or blocking Review Required findings; one or more non-blocking Recommendation or Warning findings remain. |
| 🟠 **Review required** | One or more unresolved Review Required findings exist, required metadata is incomplete, jurisdiction is unsupported, or a required administrator review has not been recorded. |
| 🔴 **High Risk** | One or more unresolved High Risk findings exist, a required retention policy is absent, a public exposure rule is breached, an overdue purge is detected, or another rule explicitly sets High Risk. |

### 9.2 Status calculation

Recommended precedence:

```text
High Risk
> Review required
> Ready with recommendations
> Ready
```

A valid authorised override may change whether a finding contributes to effective status, but the original finding remains recorded.

### 9.3 Blocking behaviour

Default Publish behaviour:

| Status | Publish behaviour |
|---|---|
| Ready | Publish allowed. |
| Ready with recommendations | Publish allowed after warnings are displayed. |
| Review required | Publish blocked for normal administrators; authorised override required. |
| High Risk | Publish blocked; only designated compliance administrators may override where the rule is marked overridable. |

Some High Risk rules must be non-overridable in MVP, including:

- no privacy page where personal data is publicly collected;
- public learner names in a public report;
- missing retention for retained personal or Special Category data;
- child-linked public media without required review;
- missing Component metadata contract.

### 9.4 Stale status

If source configuration changes after assessment, display:

`Status unavailable — rescan required.`

A stale Ready status must not permit publication.

## 10. Generated Documentation

The platform must generate documents from the current structured deployment inventory and assessment.

It must not ask administrators to rewrite detected fields, Components or retention settings.

The MVP must describe and populate the following outputs. The implementation should use structured templates and versioned clauses. This specification does not define the final legal wording.

### 10.1 Hosted Privacy Page

**Purpose:** Provide a deployment-specific privacy information page that can be linked from the participant experience.

**Structure:**

- deployment and controller identity;
- privacy/DPO contact;
- purpose of the deployment;
- categories of information processed;
- whether information is entered, generated or derived;
- how the deployment uses the information;
- processor and joint-controller details where configured;
- retention summary;
- audience-specific information where applicable;
- participant rights and contact route;
- external privacy-policy link where supplied;
- document version and last-updated date.

**Requirements:**

- generated from the current deployment configuration;
- hosted at a stable deployment-specific URL;
- versioned on publish;
- accessible before data submission;
- no internal field IDs or technical implementation language;
- clearly indicate where no directly identifying information is collected.

### 10.2 Website Snippet

**Purpose:** Give the client a short block for inclusion in an existing website privacy notice or campaign page.

**Structure:**

- name/purpose of the digital deployment;
- concise categories of data processed;
- whether data is anonymous, pseudonymous or identifiable;
- retention summary;
- link to the hosted privacy page or controller policy;
- contact route.

**Output formats:**

- copyable rich text;
- plain text;
- HTML snippet;
- downloadable `.docx` or `.txt` may be added where supported.

The snippet generator must support an explicit no-personal-data statement when the scanner confirms that configuration.

### 10.3 Client Summary

**Purpose:** Provide a concise client-facing record of how the deployment handles data.

**Structure:**

- deployment scope and purpose;
- Controller, Processor and Joint Controllers;
- audience and countries;
- detected data categories;
- public/report exposure;
- retention settings;
- current compliance status;
- unresolved recommendations;
- generated-document links;
- approval/version record.

The summary must avoid raw event lists unless included in an appendix.

### 10.4 Internal Compliance Report

**Purpose:** Provide the detailed implementation and audit record for Real Nation and authorised client reviewers.

**Structure:**

- deployment identifiers and version;
- scan timestamp and metadata version;
- complete Component and field inventory;
- automatic and effective classifications;
- enabled collection/report/publication states;
- controller and processor configuration;
- retention policies and purge schedule;
- every rule result;
- overrides and justifications;
- publish-validation result;
- documentation versions;
- known limitations and unsupported jurisdictions;
- change history.

### 10.5 Compliance Checklist

**Purpose:** Provide an actionable pre-launch task list.

**Structure:**

- mandatory unresolved items;
- recommendations;
- retention checks;
- privacy-page checks;
- report exposure checks;
- moderation checks;
- named owner and completion status;
- resolution evidence or note;
- final approval record.

Checklist items should link directly to the relevant configuration control where possible.

### 10.6 Document regeneration

When source facts change:

- mark generated documents as out of date;
- show the changed source facts;
- require regeneration before Publish;
- preserve prior published versions for audit;
- do not silently mutate a previously published document version.

## 11. Publish Validation

### 11.1 Validation sequence

Before Publish, the platform must:

1. confirm the assessment is current;
2. run the deployment scanner;
3. evaluate all active rule versions;
4. validate mandatory deployment configuration;
5. validate retention coverage;
6. validate privacy-page availability;
7. validate Reporting exposure and exports;
8. validate outstanding overrides;
9. validate generated-document freshness;
10. calculate final Compliance Status.

### 11.2 Validation result

The Publish screen must display:

- effective status;
- blocking findings;
- non-blocking recommendations;
- fields and Components affected;
- direct links to resolve each issue;
- generated-document state;
- retention state;
- last scan time;
- rule-set version.

### 11.3 Overrides

Only authorised administrators may override a blocking finding.

Every override must record:

- finding ID;
- rule ID and version;
- administrator;
- justification;
- timestamp;
- deployment version;
- expiry or review date where required;
- whether publication proceeded;
- optional supporting reference.

Overrides must be invalidated when the affected source facts materially change.

### 11.4 Audit log

Publish validation must record:

- validation start and completion time;
- source deployment version;
- status before and after validation;
- rules evaluated;
- findings opened, resolved or overridden;
- administrator who published;
- generated-document versions active at publish.

## 12. Integration

### 12.1 Measurement integration

Compliance must consume:

- enabled fields and events;
- collection state;
- identity model;
- session and participant linkage;
- data classes;
- retention policies;
- deletion support;
- aggregate-preservation behaviour.

Compliance must not create a separate measurement configuration.

Changes to Measurement must trigger compliance reassessment.

### 12.2 Reporting integration

Compliance must inspect:

- report audience;
- visible fields and metrics;
- learner-level drill-down;
- public/private access;
- exports;
- secure-link expiry;
- small-number settings;
- report retention and snapshot behaviour.

Reporting must consume Compliance status and prevent prohibited fields from appearing in public views or exports.

### 12.3 Retention integration

Compliance must:

- verify that every retained non-anonymous data class has an applicable policy;
- display inherited and overridden policies;
- show next purge date;
- receive purge success/failure state;
- flag overdue or failed purge operations;
- include retention facts in generated documents;
- retain purge audit records separately from deleted participant data.

### 12.4 Procedural documentation integration

Generated checklist and internal report outputs must link to applicable procedural records, including:

- moderation procedure;
- data-subject request procedure;
- incident escalation route;
- content-removal procedure;
- event-device handling procedure;
- access-control owner;
- client approval record.

The MVP may store links or references rather than authoring these procedures.

### 12.5 Future rule-engine integration

The rule engine must support:

- versioned rule packs;
- jurisdiction-specific rule packs;
- organisation-specific policy packs;
- rule activation dates;
- migration/re-evaluation of existing deployments;
- machine-readable rule results;
- non-AI deterministic status calculation;
- optional future AI-assisted explanation that does not alter status directly.

## 16. Implementation Acceptance Criteria

### 16.1 Scanning and inventory

The Compliance Component is acceptable when:

- it can scan a Course, Flow or standalone deployment;
- all nested deployed Components are included exactly once;
- each detected field and event identifies its source Component and instance;
- supported, enabled, collected, reported, published, retained and transient states are distinguishable;
- the scanner uses the shared metadata contract rather than a duplicate manual inventory;
- missing metadata generates a Review Required finding;
- a source change marks the assessment stale;
- rescanning updates findings without deleting audit history.

### 16.2 Classification

- every enabled field has at least one effective classification;
- classifications support all categories in Section 6;
- multiple classifications may apply to one field;
- children’s context augments participant-linked records automatically;
- automatic and overridden classifications remain separately visible;
- manual override requires justification and administrator identity;
- Special Category data triggers the required restrictions and review state.

### 16.3 Deployment configuration

- Controller, Processor, Joint Controllers and privacy contacts use structured records;
- inherited values identify their source;
- required values are validated before publish;
- audience, country and context tags are available to the rule engine;
- contradictory deployment settings generate findings;
- unsupported countries cannot receive Ready status.

### 16.4 Rule engine

- rules are deterministic and versioned;
- every MVP rule in Section 8.4 is implemented and testable;
- each finding records evaluated facts and affected scope;
- all findings remain available even when grouped in the UI;
- highest-severity precedence is applied correctly;
- changing the underlying configuration re-evaluates affected rules;
- resolved findings close automatically where the source condition no longer applies;
- overrides do not erase original findings;
- stale overrides are invalidated.

### 16.5 Status

- exactly one effective status is calculated per deployment;
- Ready is impossible with unresolved recommendations, warnings, review requirements or high-risk findings;
- Ready with recommendations is impossible with blocking Review Required or High Risk findings;
- Review Required and High Risk status block normal publication;
- stale assessments cannot publish;
- status is available to Reporting and the deployment overview.

### 16.6 Generated documentation

- the hosted privacy page is generated from current structured deployment facts;
- the website snippet can state either detected processing or confirmed no-personal-data processing;
- the client summary, internal report and checklist are generated;
- generated outputs record template and deployment version;
- documents become stale after relevant configuration changes;
- prior published versions remain available for audit;
- no generated output exposes internal IDs unless intended for the internal report;
- the generated output describes detected processing accurately without inventing fields.

### 16.7 Retention

- every retained Personal Data, Pseudonymous, Special Category, Media and participant-linked Behavioural record is covered by a policy;
- uncovered retained records generate High Risk;
- next purge date is calculated;
- purge success/failure is visible;
- overdue purges generate High Risk;
- deletion of personal records does not delete permitted anonymous aggregates;
- retention details appear in generated outputs.

### 16.8 Reporting integration

- Compliance reads report audience, visibility, drill-down and export settings;
- public learner names trigger High Risk;
- personal and technical fields are hidden from public reports by default;
- public report publication is blocked where prohibited findings remain;
- report exports respect Compliance restrictions;
- reports display current Compliance and retention status.

### 16.9 Publish validation and audit

- Publish always runs a current scan and rule evaluation;
- blocking findings are displayed with direct resolution links;
- only authorised administrators may override;
- every override records the required audit fields;
- non-overridable rules cannot be bypassed;
- validation records the deployment version and active rule set;
- the published deployment records the active generated-document versions;
- configuration changes after publication create a new pending assessment rather than rewriting the historic publication record.

### 16.10 UI and permissions

- all recommended sections in Section 13 are implemented or represented in an equivalent accessible navigation model;
- status is communicated with text as well as colour/icon;
- administrators are not asked to manually describe known fields or events;
- inherited and detected values show their source;
- findings link to the relevant configuration control;
- only authorised roles can change organisation records, classifications, overrides or publish decisions;
- client viewers cannot access internal findings, raw identifiers or override records unless explicitly permitted.
