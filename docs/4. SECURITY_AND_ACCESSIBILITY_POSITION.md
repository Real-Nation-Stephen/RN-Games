This document describes **what the platform does today**, **what is planned next**, **what is explicitly deferred**, and **known gaps**. It is a position paper, not a certification, audit report, or legal opinion.

---

## Executive summary

RN Game Studio is a Netlify-hosted monorepo with a **trusted-designer Studio** (authenticated) and **unauthenticated public play** surfaces. Security today is pragmatic: HTTPS, Identity-gated admin APIs, coarse access secrets on some live-session features, and basic response headers. Accessibility today is **component-local and inconsistent**: some player surfaces include ARIA, focus, and reduced-motion handling; there is no platform-wide WCAG programme, automated checking, or accessibility gate before publish.

Measurement, reporting, and compliance pilots add Postgres event storage and advisory privacy scanning. Those layers improve **data governance visibility** but do not yet enforce publish blocking, purge jobs, rate limits, or accessibility validation.

Use this document as the factual baseline when drafting a formal Security specification and Accessibility specification.

# Part A ŌĆö Security

## A.1 Current approach

### Platform and trust model

| Layer | Model |
|-------|--------|
| **Hosting** | Netlify (HTTPS by default on `*.netlify.app` and custom domains) |
| **Studio (admin)** | Netlify Identity widget; protected APIs require a Bearer JWT or Lambda `clientContext.user` |
| **Public play** | No login; access by slug, room code, or opaque token |
| **Tenancy** | Single Studio account ŌĆö any authenticated Identity user can access the full admin surface |
| **Uploaders** | Trusted RN designers only; no public asset upload or cartridge sideload (see [PLANNING.md](./PLANNING.md)) |

### Authentication and authorization (implemented)

**Studio APIs** (`requireAuth` in `netlify/functions/lib/auth.mjs`):

- Accept `Authorization: Bearer <JWT>` or Netlify `context.clientContext.user`.
- Return **401** when no user is present.
- **No role-based access control (RBAC)** ŌĆö no per-client, per-project, or per-game permissions.
- JWT handling **decodes the payload** (`sub`, `email`) but does **not cryptographically verify the token signature** in application code. In practice, Netlify Identity + function context is the primary trust path; treat direct Bearer usage as a known hardening opportunity.

**Public session controls (partial):**

| Feature | Control |
|---------|---------|
| Quiz host actions | `hostKey` required on `POST /api/quiz-control` |
| Leaderboard moderation | `moderatorPin` on moderator surface (default `1234` in shared defaults ŌĆö designers should change) |
| Course / experience draft preview | `previewToken` (UUID) on unpublished content |
| Course resume | Short opaque `resumeToken` in URL (`?resumeToken=`) mapped via blob index |
| Published content | Slug-based ŌĆö treat as **unlisted**, not secret |

**Local development:** `VITE_DEV_AUTH=1` bypasses Identity with a fixed dev JWT. Must never be enabled in production builds.

### API exposure model

| Class | Examples | Auth |
|-------|----------|------|
| Studio CRUD | `/api/wheels`, `/api/experiences`, `/api/courses`, `/api/upload` | Identity JWT |
| Studio analytics / M&R | `/api/track-stats`, `/api/deployment-measurement`, `/api/measurement-replay` | Identity JWT |
| Public config | `/api/public-wheel`, `/api/public-experience`, `/api/public-course` | None (slug + optional preview token) |
| Public session | `/api/course-session`, `/api/experience-session`, `/api/quiz-join`, pinboard state | None / room code |
| Public ingest | `POST /api/track`, `POST /api/log-spin` | None |
| Public assets | `GET /api/file?id=` | None if ID is known |

Most API responses include `Access-Control-Allow-Origin: *`.

### Transport and browser headers

Configured in `netlify.toml`:

| Header | Value |
|--------|-------|
| `X-Frame-Options` | `SAMEORIGIN` |
| `X-Content-Type-Options` | `nosniff` |

**Not implemented:** Content-Security-Policy (CSP), `Strict-Transport-Security` explicit config (rely on Netlify), `Referrer-Policy`, `Permissions-Policy`.

### Data storage and secrets

| Store | Contents | Notes |
|-------|----------|-------|
| Netlify Blobs | Game configs, sessions, uploads, track-log buffer | Same-origin file URLs; eventual consistency |
| Netlify Database (Postgres) | Measurement pilot: `events`, `measurement_config`, `ingest_replay_runs` | Connection via `NETLIFY_DB_URL` / `@netlify/database` |
| Google Sheets (optional) | Spin reporting rows | Service account env vars; spreadsheet shared with SA |
| Client | Session IDs, resume tokens in URL, local progress | See [COMPONENT_DATA_EVENT_INVENTORY.md](./COMPONENT_DATA_EVENT_INVENTORY.md) |

Uploaded binaries: Studio-authenticated, max **~4 MB** per file (`netlify/functions/upload.mjs`). Served from `/api/file` with long cache headers. **No authentication on file fetch** ŌĆö security relies on unguessable IDs.

### Input handling and XSS

- **Course shell** and several player surfaces use `escapeHtml()` when injecting user- or config-derived strings into templates.
- **Page modules** (form, landing, etc.) generally build DOM with `createElement` / `textContent` rather than raw `innerHTML` for field values.
- **Designer-authored copy** (headlines, labels, HTML blocks on landing pages) is rendered as configured ŌĆö trusted-designer model, not a sanitisation sandbox.
- **No CSP** to constrain inline script or third-party embeds. Landing iframe/embed blocks are a future concern.

### Abuse and operational security

| Control | Status |
|---------|--------|
| Rate limiting on public endpoints | **Not implemented** (locked preference: implement ŌĆö [ROADMAP_QUESTIONS.md](./ROADMAP_QUESTIONS.md) T-4) |
| Audit log of admin actions | **Not implemented** |
| Per-client data isolation | **Not implemented** |
| Automated purge / retention enforcement | **Not implemented** (retention days configured in M&R UI; no purge job) |
| Publish blocking on compliance findings | **Not implemented** (advisory pilot only) |
| IP logging / retention policy | **Not implemented** as first-class control |
| DDoS / WAF | Rely on Netlify platform defaults |

### Measurement and compliance security (pilot)

- `POST /api/track` is **public** ŌĆö accepts canonical events, writes to Blobs + Postgres with idempotent `event_id`. Suitable for client-side beacons but **abusable for spam** without rate limits.
- `POST /api/measurement-replay` is Studio-authenticated ŌĆö replays blob track-log into DB.
- Compliance scan and doc generation are **configuration-based**; generated privacy text is advisory and must be legally reviewed before publication.

### GDPR and consent (security-adjacent)

- **Pin board** has structured consent UI before guest submit.
- **Consent** and **email-signup** page modules exist; opt-in quality varies by configuration.
- **Universal consent layer** across all game types is not built.
- Formal DPIA / privacy sign-off process is tracked in [ANALYTICS_LEGAL_QUESTIONNAIRE.md](./ANALYTICS_LEGAL_QUESTIONNAIRE.md) ŌĆö not automated in platform.

### Environment separation

- **Production:** `rn-games.netlify.app`
- **Staging:** Locked preference to maintain separate environment ([ROADMAP_QUESTIONS.md](./ROADMAP_QUESTIONS.md) T-5) ŌĆö operational discipline, not enforced in codebase.

## A.2 Planned improvements

These align with existing roadmap, questionnaire answers, and pilot work already in flight. Priority order is indicative ŌĆö final sequencing belongs in the Security specification.

| Area | Intent |
|------|--------|
| **Rate limiting** | Throttle public write endpoints (`/api/track`, `/api/log-spin`, session creation, pinboard submit, quiz join) per IP / fingerprint |
| **JWT verification** | Verify Identity JWT signature (or rely exclusively on Netlify `clientContext` and reject unverified Bearer tokens) |
| **Staging environment** | Separate Netlify site + blob store + database branch for pre-production testing |
| **Retention purge jobs** | Scheduled deletion aligned with `measurement.retention.defaultDays` and compliance rules (`PURGE_OVERDUE_001`) |
| **Secrets rotation runbook** | Documented rotation for `NETLIFY_DB_URL`, Google SA, preview/resume token compromise |
| **Upload hardening** | MIME allowlist, image re-encode, optional virus scan hook for untrusted upload paths (future) |
| **CORS tightening** | Restrict admin mutations to known Studio origins where embed is not required |
| **Compliance publish gate** | Block or warn on publish when indicative scan is Review Required / High Risk (per full Compliance spec) |
| **Audit trail** | Record publish, compliance override, measurement config change, and export actions |
| **RBAC (phased)** | Roles for admin vs analytics-only vs client viewer ŌĆö prerequisite for client-facing dashboards |

## A.4 Known gaps (security)

Factual gaps as of July 2026. Severity is **platform risk context**, not a formal CVSS assessment.

| Gap | Impact | Current mitigation |
|-----|--------|------------------|
| **No rate limiting** | Abuse, cost inflation, DB/blob spam via `/api/track` and session APIs | Netlify platform defaults only |
| **JWT not signature-verified in app code** | Forged Bearer token could work if endpoint relies on decode-only path | Netlify function context for same-origin Studio |
| **Single-tenant admin** | Any Studio user sees all clientsŌĆÖ content | Trust model: RN staff only |
| **Slug = capability** | Anyone with slug can load published config | Obscurity; no DRM |
| **File URLs are guessable IDs** | Direct asset access if ID leaks | Opaque IDs; designer-only upload |
| **Public track ingest** | Event pollution, storage cost | Idempotent `event_id`; no auth |
| **Default moderator PIN** | Weak live-session protection if unchanged | Designer must set PIN in editor |
| **Resume / preview tokens in URL** | Token leakage via referrer, screenshots, shared links | Opaque tokens; HTTPS |
| **CORS `*`** | Cross-origin browser calls to APIs from any site | Mostly read/config; writes still need session secrets where applicable |
| **No CSP** | XSS from designer HTML/embed content has full script capability | Trusted designer model |
| **No purge enforcement** | Personal data may outlive configured retention | Manual ops; compliance advisory only |
| **No audit log** | Cannot reconstruct who changed what | Git history for code only |
| **Sheets credentials in env** | Key compromise exposes reporting spreadsheet | Netlify secret storage; SA least-privilege |
| **Compliance docs advisory** | Generated privacy text may be wrong or incomplete | Human legal review required |
| **Edge router transient 500s** | Availability, not confidentiality | Retries / fallback URLs ([INFRASTRUCTURE.md](./INFRASTRUCTURE.md)) |

# Part B ŌĆö Accessibility

## B.1 Current approach

### Policy and process

- **No platform-wide accessibility policy** or WCAG conformance target is encoded in the repository.
- **No automated accessibility testing** in CI/CD or pre-publish checks.
- **No accessibility section** in the Compliance pilot UI (future: Compliance spec ┬¦15.2).
- **Designer responsibility** for colour contrast, motion, and copy clarity when configuring per-component hex colours and animations.

### Player surfaces (public)

**Baseline HTML**

- Player pages set `<html lang="en">` across game and page-module entry HTML.
- Viewport meta present on standard templates.

**Patterns in use (varies by component)**

| Pattern | Where present (examples) |
|---------|--------------------------|
| `aria-label` on controls | Wheel spin zone, scratcher canvas, flip-card slots, quiz fullscreen, runner fullscreen |
| `aria-pressed` | Fullscreen toggles, mute buttons |
| `aria-hidden` | Decorative layers, preload iframes |
| `aria-live` | Pin board canvas updates |
| `role="dialog"` / `aria-modal` | Orientation gate, pinboard clear dialog |
| `label` + `htmlFor` | Form page module field rendering |
| `:focus-visible` styles | Wheel styles, flip-cards, runner character select |
| `prefers-reduced-motion` | Wheel/scratcher CSS, flip-cards, fanfare helper, some animations skipped in JS |

**Landing page module**

- Block editor includes **Title (accessibility)** for image alt text ([LandingBlocksEditor.tsx](../packages/admin/src/components/LandingBlocksEditor.tsx)).

**Course shell**

- Partial semantic structure; some decorative emoji marked `aria-hidden`.
- Progress and item lists are DOM text ŌĆö not fully tested with screen readers.

### Studio (admin)

- Desktop-first React SPA; usable on large screens, **not optimised for mobile editing**.
- Sparse ARIA usage (search, duplicate actions, colour picker labels, flow toolbar separators).
- Collapsible sections (`CollapsibleSection`) use button toggles ŌĆö keyboard operable, but **no skip links** or landmark navigation strategy documented.
- React Flow experience canvas ŌĆö accessibility of node editing not audited.

### Game-specific realities

| Component type | Accessibility characteristic |
|----------------|------------------------------|
| **Spinning wheel, scratcher, catch, runner** | Canvas / game-loop interaction; limited non-visual alternatives |
| **Flip cards** | Better keyboard focus on card slots in current build |
| **Quiz** | Host/present/join flows; mobile coverage better than wheel |
| **Pin board** | Photo upload and visual board ŌĆö moderation required for UGC |
| **Certificate / badge** | Visual reward; PDF/PNG output accessibility not validated |
| **Orientation gate** | Blocks portrait on some games ŌĆö **barrier** for users who cannot rotate device |

### Relationship to compliance and privacy

- Compliance pilot can flag **free-text fields**, **email collection**, and **leaderboard names** ŌĆö privacy risks, not WCAG failures.
- Accessible privacy notice delivery (plain language, before submit) is partially supported via consent and form modules but **not validated** against WCAG or EN 301 549.

## B.4 Known gaps (accessibility)

| Gap | Impact | Current state |
|-----|--------|---------------|
| **No WCAG conformance claim** | Client/procurement risk | None stated |
| **Inconsistent ARIA** | Screen reader experience varies | Per-component ad hoc |

| **Certificate PDF** | May lack tags/reading order | Not tested |
| **UGC (pinboard photos)** | Alt text not meaningful | Moderation workflow |
| **Compliance docs** | Do not include accessibility statement | Privacy-only pilot |

# Part C ŌĆö Cross-cutting dependencies

Security and accessibility intersect in several areas that should stay aligned across specifications:

| Topic | Security | Accessibility |
|-------|----------|---------------|
| **Public forms** | PII handling, retention, consent evidence | Labels, errors, keyboard, readable privacy links |
| **Resume / magic links** | Token secrecy, expiry | Link text clarity, session restore messaging |
| **Preview tokens** | Unpublished content exposure | Preview banner for assistive tech |
| **Reporting exports** | AuthZ on CSV / identifiable exports | Export format usability |
| **Publish workflow** | Compliance gate, audit | Accessibility gate (deferred) |
| **Rate limiting** | Abuse prevention | Legitimate assistive-tech traffic patterns |
