# Tracking & Measurement Specification

**Platform:** Real Nation Digital Platform  
**Status:** Draft v1 for implementation planning  
**Source of truth:** `COMPONENT_DATA_EVENT_INVENTORY.md`  
**Scope:** Tracking architecture, event taxonomy, data classes, default measurement, context scoping, storage, aggregation, retention hooks, and the shared metadata contract used by Reporting and Compliance.

---

## 1. Purpose

This specification defines how the Real Nation Digital Platform should record, structure and retain activity across Courses, Flows and Components.

It is intended to ensure that:

- tracking is consistent across the platform;
- the same fact is defined once and reused by Measurement, Reporting and Compliance;
- activity can be attributed correctly to standalone Components, Flows and Courses;
- useful client-facing metrics can be generated without collecting unnecessary data;
- identifiable, pseudonymous and aggregate data are kept distinct;
- retention and deletion can be applied reliably;
- new Components can participate in tracking through a standard metadata contract rather than bespoke analytics code.

This specification does **not** authorise any particular processing activity. Legal bases, privacy wording, consent requirements and best-practice warnings are defined in the separate Compliance Component Specification.

## 2. Platform hierarchy and terminology

The current hierarchy is:

```text
Course
└── Flow
    └── Component
```

A Component may also be deployed directly without a parent Flow or Course.

The following terms are used throughout this document:

| Term | Meaning |
|---|---|
| **Course** | A structured learning or participation programme containing one or more items, which may include Flows and Components. |
| **Flow** | An ordered journey containing multiple Components. |
| **Component** | A reusable participant-facing or foundation building block, such as a landing page, form, game, quiz, badge or certificate. |
| **Deployment** | A specific use of a Course, Flow or Component in a given context. |
| **Measurement Component** | A foundation Component attached to a Course, Flow or standalone Component that defines what is collected, aggregated and made available to reporting. |
| **Compliance Component** | A foundation Component that interprets the same configuration and metadata for privacy, retention, best-practice and documentation purposes. |
| **Session** | A bounded period of participant activity within a deployment. |
| **Learner/Profile** | A persistent or resumable participant record where identity or progress is retained across sessions. |

The word **Experience** should not be used in the product-facing architecture. Existing code or data keys using `experience` may remain temporarily during migration but should be treated as legacy terminology.

## 3. Core design principles

### 3.1 Define once, reuse everywhere

Every Component must expose one shared metadata contract describing:

- events it can emit;
- fields it can collect or derive;
- metrics it can support;
- whether those fields are anonymous, pseudonymous, personal or operational;
- which contexts it supports;
- which retention and compliance rules apply.

Tracking, Reporting and Compliance must consume this same source of truth.

### 3.2 Tracking should happen by default, not by memory

A Component should emit its standard operational events automatically whenever it is used. Project teams should not need to remember to add custom analytics code to every build.

### 3.3 Collecting and reporting are separate decisions

For each field or event, the platform must distinguish between:

1. **Supported** — the platform is technically capable of collecting it;
2. **Collect** — it is enabled for this deployment;
3. **Show in report** — it appears in a client-facing or internal dashboard;
4. **Retain at identifiable level** — it remains linked to a session or learner for the configured period.

A field may be needed internally but hidden from dashboards. A metric may be shown only as an aggregate after the identifiable source records have been deleted.

### 3.4 Scope must be explicit

The same Component can appear in multiple contexts. A Dino Runner used in Digital 404 must not be mixed with the same Runner accessed directly or used in another client campaign.

Every event must therefore carry an explicit deployment context.

### 3.5 Personal and aggregate data must remain separable

The platform must allow personal or pseudonymous records to be deleted while retaining genuinely anonymous aggregate statistics where appropriate.

### 3.6 Data minimisation is the default

The platform should collect the minimum useful dataset by default. Optional diagnostics and high-granularity interactions should be disabled unless they serve a clear purpose.

### 3.7 Instrumentation must support reliable measurement

A metric should not appear in a dashboard unless the source events required to calculate it are reliably emitted and attributable.

## 4. Recommended storage direction

### 4.1 Decision

Implement the new tracking model in a relational SQL database rather than extending the current Netlify Blobs event store.

Netlify Blobs may remain appropriate for:

- static configuration;
- generated assets;
- simple cached outputs;
- temporary migration support;
- low-frequency unstructured records where relational querying is not required.

SQL should become the system of record for:

- deployments;
- sessions;
- participants/profiles;
- events;
- event properties;
- component outcomes;
- consent and compliance records;
- retention schedules;
- purge jobs;
- aggregate metrics;
- report definitions.

### 4.2 Why move now

The platform is about to formalise its event model, reporting, retention and compliance behaviour. Moving before that structure is deeply embedded avoids recreating the same work later.

SQL is better suited to:

- filtering by Course, Flow, Component and deployment;
- joining session, learner, event and outcome data;
- calculating funnels, return rates and averages;
- applying retention by date and data class;
- locating and deleting a learner or session;
- distinguishing direct and nested use of the same Component;
- producing reusable report views;
- auditing changes and purge operations.

### 4.3 Migration approach

The implementation should not require a single all-at-once migration.

Recommended sequence:

1. Create the canonical SQL schema.
2. Route all new canonical events to SQL.
3. Continue writing legacy events to Blobs temporarily only where required for compatibility.
4. Build an import script for valuable historic Blob data.
5. Validate parity between legacy and SQL reports.
6. retire Blob-based tracking after sign-off.

Do not import data simply because it exists. Historic data should be reviewed for usefulness, consistency and compliance before migration.

## 5. Measurement Component

### 5.1 Role

The Measurement Component is a non-learner-facing foundation Component that can be attached to:

- a Course;
- a Flow;
- a standalone Component deployment.

It defines:

- the scope of activity to listen to;
- which supported events and fields are collected;
- which metrics are derived;
- which labels and groupings are available to reporting;
- whether data is stored only as aggregates or linked to sessions/learners;
- retention settings inherited by the attached scope;
- a relationship to the attached Compliance Component.

### 5.2 Scope behaviour

If attached to a Course, the Measurement Component listens to activity generated:

- by the Course shell;
- within every Flow nested in that Course;
- within every Component nested in those Flows;
- by Components added directly to the Course.

If attached to a Flow, it listens only to:

- the Flow shell;
- Components used within that Flow;
- activity generated while those Components are accessed through that Flow.

If attached to a standalone Component, it listens only to direct use of that deployment.

### 5.3 Scope precedence

A single participant action may occur within a Component nested inside both a Flow and a Course. The event should be recorded once with all relevant context IDs, not duplicated for each parent.

The event may then be included in multiple report scopes through SQL relationships.

### 5.4 Inheritance

Recommended inheritance order:

```text
Platform defaults
→ Course Measurement settings
→ Flow overrides
→ Component deployment overrides
```

Overrides should be explicit, visible and auditable.

## 6. Canonical context model

Every event must contain a standard context object.

```json
{
  "deploymentId": "dep_...",
  "deploymentContext": "course",
  "courseId": "course_...",
  "courseSessionId": "cs_...",
  "courseItemId": "item_...",
  "flowId": "flow_...",
  "flowSessionId": "fs_...",
  "flowStepId": "step_...",
  "componentType": "runner",
  "componentId": "component_...",
  "componentInstanceId": "instance_...",
  "componentVersionId": "version_...",
  "sessionId": "session_...",
  "participantId": "participant_...",
  "occurredAt": "2026-07-12T12:00:00.000Z"
}
```

Only fields relevant to the current context need values, but the schema must remain stable.

### 6.1 Required context fields

Every event requires:

- `eventId`
- `eventName`
- `occurredAt`
- `deploymentId`
- `deploymentContext`
- `componentType`
- `componentInstanceId`
- `sessionId`

### 6.2 Conditional context fields

Where applicable:

- `courseId`
- `courseSessionId`
- `courseItemId`
- `flowId`
- `flowSessionId`
- `flowStepId`
- `participantId`
- `classId`
- `cohortId`
- `campaignId`
- `locationId`
- `versionId`

### 6.3 Deployment context values

Use a controlled enum:

- `standalone_component`
- `flow`
- `course`
- `course_flow`
- `preview`
- `admin_test`

Preview and admin-test activity must be excluded from live client reporting by default.

## 7. Canonical event model

### 7.1 Event envelope

```json
{
  "eventId": "evt_...",
  "schemaVersion": "1.0",
  "eventName": "runner.round_completed",
  "eventCategory": "engagement",
  "occurredAt": "2026-07-12T12:00:00.000Z",
  "context": {},
  "properties": {},
  "privacy": {
    "dataClass": "B",
    "containsPersonalData": false,
    "containsPseudonymousData": true
  }
}
```

### 7.2 Naming convention

Use:

```text
<object>.<past-tense_action>
```

Examples:

- `course.viewed`
- `course.started`
- `course.completed`
- `course.item_started`
- `course.item_completed`
- `flow.started`
- `flow.step_started`
- `flow.step_completed`
- `flow.completed`
- `component.viewed`
- `component.started`
- `component.completed`
- `runner.round_started`
- `runner.round_completed`
- `wheel.spin_completed`
- `form.submitted`
- `certificate.downloaded`

Legacy names such as `experience.step_start` should be mapped to the canonical vocabulary during migration.

### 7.3 Event categories

| Category | Purpose |
|---|---|
| **Exposure** | The participant was shown a Course, Flow, Component or content item. |
| **Engagement** | The participant actively interacted with it. |
| **Progression** | Started, completed, resumed, abandoned or moved between stages. |
| **Learning** | Answer, score, confidence, recall or outcome data. |
| **Game performance** | Game start, finish, score, distance, survival, pickups or obstacles. |
| **Conversion** | CTA click, signup, form submission, redemption, download or external handoff. |
| **Recognition** | Badge or certificate earned/downloaded. |
| **Consent/compliance** | Notice shown, preference recorded, consent changed or withdrawn. |
| **Operational** | Errors, failed loads, latency and technical diagnostics. |

### 7.4 Universal lifecycle events

Every participant-facing Component should support, where meaningful:

- `component.viewed`
- `component.started`
- `component.engaged`
- `component.completed`
- `component.abandoned`
- `component.replayed`
- `component.error_encountered`

Not every event must fire for every Component. For example, a simple informational page may use `viewed`, `engaged` and `completed`, while a game uses all lifecycle events.

## 8. Data classification for measurement

This specification uses five measurement classes. The Compliance Specification will add legal treatment and rule logic.

| Class | Name | Description | Typical examples |
|---|---|---|---|
| **A** | Anonymous aggregate | Cannot reasonably be traced to a person or session. | Total plays, aggregate average score, count by day. |
| **B** | Pseudonymous/session-level | Linked to an opaque session or participant ID and may still be personal data. | Session ID, replay history, per-session score. |
| **C** | Ordinary personal data | Directly identifies or is clearly linked to an identifiable person. | Email, name, linked progress, certificate record. |
| **D** | Higher-risk/context-sensitive | Requires heightened review due to subject matter, audience or content. | Children’s data, special-category answers, open-text disclosures, uploaded photos. |
| **E** | Operational/technical | Used for security, reliability or debugging rather than client performance reporting. | Error logs, rate-limit keys, request traces. |

Each field and event must declare a class in the metadata contract.

### 8.1 Important distinction

A generated ID is not automatically anonymous. If it can be linked to a learner, email, class, session history or other record, it must be treated as Class B or above.

## 9. Collection states

Every supported field/event must support the following configuration states:

| Setting | Meaning |
|---|---|
| `supported` | Technically available for this Component type. |
| `collect` | Enabled for this deployment. |
| `report` | Available in dashboards and exports. |
| `retainIdentifiable` | Retained with participant/session linkage. |
| `aggregateAfterExpiry` | Contributes to anonymous aggregates after source deletion. |
| `requiredForOperation` | Cannot be disabled without breaking the Component. |
| `requiredForSecurity` | Collected for abuse prevention or system security. |

A typical configuration may be:

| Data item | Supported | Collect | Report | Retain identifiable | Aggregate after expiry |
|---|---:|---:|---:|---:|---:|
| Session ID | Yes | Yes | No | Yes | No |
| Email | Yes | Yes | No | Yes | No |
| Quiz score | Yes | Yes | Yes | Yes | Yes |
| Average quiz score | Derived | Automatic | Yes | No | Yes |
| Character choice | Yes | Yes | Yes | Optional | Yes |
| Error trace | Yes | Yes | No | Short-term | No |

## 10. Default tracking baseline

The platform should begin with a conservative but useful baseline.

### 10.1 Track by default

For all deployments:

- views;
- starts;
- completions;
- completion status;
- timestamp;
- duration;
- deployment context;
- Component type and instance;
- Flow/Course attribution where relevant;
- anonymous or pseudonymous session ID;
- essential game or quiz outcome;
- technical errors required to maintain the service.

### 10.2 Track only when enabled or required

- email;
- name;
- phone number;
- date of birth;
- class or teacher association;
- exact free-text answers;
- uploaded media;
- public leaderboard alias;
- exact answer-by-answer quiz history;
- detailed clickstream;
- referral and UTM data;
- location beyond a deliberately selected campaign/location value;
- long-term cross-course identity.

### 10.3 Do not track by default

- pointer movement;
- every scroll increment;
- device fingerprinting;
- precise location;
- unnecessary user-agent detail;
- raw IP addresses beyond short-lived security needs;
- invisible interaction telemetry without a defined use case;
- duplicate copies of the same form field in multiple storage paths.

## 11. Session, learner and identity model

### 11.1 Session

A session represents one bounded participation period in a specific deployment.

Recommended fields:

- `sessionId`
- `deploymentId`
- `startedAt`
- `lastActivityAt`
- `completedAt`
- `status`
- `participantId` where present
- `courseSessionId` or `flowSessionId` where present
- `entrySource` where deliberately tracked

### 11.2 Participant

A participant is an opaque identity used to link activity within the configured scope.

It should not automatically become a global identity across all clients or Courses.

Recommended default scopes:

- standalone deployment;
- Course;
- class/cohort;
- client account only where explicitly required.

### 11.3 Learner profile

A learner profile exists when a Course supports returning progress, email login, class enrolment, badges or certificates.

The profile may include:

- participant ID;
- resume token;
- email, where enabled;
- display name, where enabled;
- Course progress;
- earned badges;
- earned certificates;
- class/cohort association;
- relevant outcomes.

### 11.4 Resume tokens

Resume tokens are access credentials and must not be treated as harmless analytics values.

They should:

- be stored securely;
- never appear in reports;
- be redacted from logs;
- be revocable;
- expire or be deleted with the related learner profile;
- not be used as a general-purpose participant ID.

### 13.1 Required Flow events

- `flow.viewed`
- `flow.started`
- `flow.step_viewed`
- `flow.step_started`
- `flow.step_engaged`
- `flow.step_completed`
- `flow.resumed`
- `flow.completed`
- `flow.abandoned`

### 13.2 Core Flow metrics

- total starts;
- unique sessions;
- completion rate;
- average completion time;
- progression rate between steps;
- drop-off by step;
- average time per step;
- replay rate;
- outcome distribution;
- CTA conversion to the next step;
- error or failed-load rate.

### 13.3 Completion logic

The current overlapping completion signals must be consolidated into one canonical completion contract.

Each child Component should return:

```json
{
  "status": "completed",
  "completedAt": "...",
  "outcomes": {},
  "completionReason": "participant_action"
}
```

The Flow shell should then emit one canonical `flow.step_completed` event and update session state once.

Legacy flags such as `courseLastStep`, `moduleItemComplete`, `flowCompleteOverride` and multiple postMessage completion variants should be mapped into the canonical contract and then retired.

## 15. Derived metric definitions

Metrics must use consistent definitions across reports.

| Metric | Definition |
|---|---|
| **Views** | Count of valid `viewed` events excluding preview/admin traffic. |
| **Unique sessions** | Distinct session IDs within the selected scope and period. |
| **Unique learners** | Distinct participant/profile IDs within the configured identity scope. Must not be used where identity is unavailable. |
| **Start rate** | Starts ÷ views. |
| **Completion rate** | Completed unique sessions ÷ started unique sessions. |
| **Engagement rate** | Unique sessions with a meaningful interaction ÷ unique sessions that viewed. |
| **Drop-off rate** | Started sessions that did not complete within the defined session window ÷ started sessions. |
| **Replay rate** | Participants/sessions with more than one valid start ÷ participants/sessions with at least one start. |
| **Return rate** | Participants who return in a later session ÷ participants with at least one session. Requires persistent participant identity. |
| **Average duration** | Mean elapsed time between valid start and completion/end events. |
| **Median duration** | Median elapsed time between valid start and completion/end events. |
| **CTA click-through rate** | Unique CTA clickers ÷ unique page viewers. |
| **Average score** | Mean valid final score per completed attempt. |
| **Highest score** | Maximum valid final score within the selected scope. |
| **Answer accuracy** | Correct submitted answers ÷ scored submitted answers. |
| **Redemption rate** | Successful redemptions ÷ eligible outcomes or redemption starts, as labelled. |
| **Badge attainment** | Learners earning the badge ÷ eligible learners. |
| **Certificate attainment** | Learners issued the certificate ÷ eligible learners. |
| **Certificate download rate** | Downloads ÷ certificates generated. |

The denominator must always be visible or documented in the Reporting Component to avoid misleading interpretation.

## 16. Duration and abandonment rules

### 16.1 Duration

Duration should be derived from server-received timestamps where possible.

Exclude:

- sessions left open indefinitely;
- preview/admin traffic;
- duplicate start/completion events;
- durations above a configurable outlier threshold.

### 16.2 Abandonment

A session may be considered abandoned when:

- it has started;
- it has not completed;
- no activity has occurred within the configured timeout;
- it has not later resumed and completed.

Recommended default timeout:

- simple Component: 30 minutes;
- Flow: 24 hours;
- Course item: 7 days;
- Course: do not mark the entire Course abandoned solely due to inactivity; use inactive/progress status instead.

These defaults must be configurable.

## 17. Retention architecture hooks

The Compliance Specification will define permitted/default periods. The Tracking system must provide the technical controls.

### 17.1 Retention inheritance

```text
Platform default
→ Client/account default
→ Course
→ Flow
→ Component deployment
→ Specific data-class override
```

### 17.2 Monthly purge batches

Retention should be enforceable through monthly batches.

Recommended model:

- each record receives `retentionBucket`, e.g. `2026-12`;
- the purge date is calculated from the configured maximum retention period;
- records in the same monthly bucket are reviewed and purged together;
- the effective retention period must never exceed the configured maximum;
- the admin view shows upcoming purge batches.

Example:

> Data collected during December 2026 under a 12-month maximum retention rule is assigned a purge date no later than 1 December 2027, according to the agreed bucket calculation.

The final boundary logic should be confirmed during legal review and implementation testing.

### 17.3 Record lifecycle

```text
Active
→ Expired
→ Pending purge
→ Deleted or anonymised
```

Recommended fields:

- `retentionPolicyId`
- `retentionClass`
- `retentionBucket`
- `expiresAt`
- `purgeScheduledAt`
- `purgedAt`
- `purgeAction`
- `legalHold` where supported

### 17.4 Aggregate preservation

After source records expire, the platform may retain only aggregates that are genuinely no longer linkable to a person or session.

The purge job should support:

- delete source event;
- delete or sever participant linkage;
- delete direct personal fields;
- preserve approved aggregate counters;
- record an audit log of the purge action.

---

## 18. Data deletion and export requirements

The new SQL system must support:

- locating all records for a participant/profile;
- locating all records for a session;
- locating all records for a deployment;
- exporting data for a selected deployment and date range;
- deleting or anonymising a participant’s linked records;
- deleting a Course or Flow deployment without leaving orphaned personal data;
- deleting form submissions independently where required;
- deleting leaderboard names while preserving approved aggregate scores;
- tracking purge and deletion actions in an audit table.

A single participant deletion must be orchestrated across:

- Course sessions;
- Flow sessions;
- events;
- form submissions;
- leaderboard entries;
- certificates and badges;
- email/signup records;
- redemption records;
- consent records;
- class/cohort relationships.

---

## 19. Data quality and deduplication

### 19.1 Idempotency

Every event must have a unique `eventId`. Re-sent events with the same ID must not create duplicates.

### 19.2 Completion deduplication

Only one canonical completion event should count for a given attempt.

### 19.3 Attempt identity

Games and quizzes should include `attemptId` so multiple plays in one session can be measured accurately.

### 19.4 Client clock handling

Store server receipt time as well as client occurrence time where offline or delayed events are possible.

### 19.5 Schema validation

Events that do not match the registered Component contract should be rejected, quarantined or logged as invalid rather than silently stored as free-form payloads.

## 24. Acceptance criteria

The Tracking & Measurement implementation is ready for pilot use when:

- every event is attributable to the correct deployment context;
- Course, Flow and standalone activity can be separated reliably;
- every shipped Component has a registered metadata contract;
- all core lifecycle events are validated and deduplicated;
- collection and report visibility are independently configurable;
- personal, pseudonymous, aggregate and operational data can be distinguished;
- retention policies can be assigned and inherited;
- purge dates can be calculated and surfaced;
- a participant/session can be located and deleted;
- preview/admin traffic is excluded by default;
- reports can be generated without exposing internal identifiers;
- the Compliance Component can inspect the exact same measurement configuration without asking for the information again.

---

## 25. Non-goals for v1

The first implementation does not need to include:

- cross-client identity;
- predictive analytics;
- automated behavioural profiling;
- branching analytics;
- real-time AI insight generation;
- device fingerprinting;
- highly granular clickstream replay;
- a universal data warehouse beyond the platform’s own reporting needs;
- permanent retention of raw event-level data.

These can be considered later only where they serve a clear product need and pass compliance review.

## 26. Final architecture summary

```text
Every Course, Flow and Component
        │
        └── exposes one shared metadata contract
                    │
                    ├── emits canonical events
                    │
                    ├── Measurement Component decides what is collected
                    │
                    ├── Reporting Component decides what is shown
                    │
                    └── Compliance Component decides what it means

Canonical events
        ↓
SQL event and session store
        ↓
Validated outcomes and aggregates
        ↓
Retention, deletion and reporting
```

The core rule is simple:

> **The platform should know what happened, where it happened, why it is being retained, and whether it should appear in a report — without requiring the same information to be entered twice.**
